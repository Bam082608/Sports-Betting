## üéØ System Overview

### Purpose

Team DNA files serve as comprehensive betting intelligence repositories containing:
- **Player Tiers:** Tier 1 (elite) and Tier 2 (high-volume) player identification
- **Team Systems:** Offensive and defensive structure analysis
- **Betting Strategies:** Team-specific betting approaches and edges
- **Situational Factors:** Home/away splits, B2B performance, travel impact
- **Goalie Intelligence:** Starting goalie analysis and backup assessment
- **Historical Patterns:** Documented betting patterns and tendencies

### File Naming Convention

```
[city-name]-[team-name].md

Examples:
- anaheim-ducks.md
- boston-bruins.md
- new-york-rangers.md (not "ny-rangers")
- utah-hockey-club.md
```

**Format:** Lowercase with hyphens, full city names

## üìñ Usage Guidelines

### Pre-Game Research Workflow

**Step 1: Identify Tonight's Games**
```
1. Check daily-logs or daily-intel for today's slate
2. Note which teams are playing
```

**Step 2: Pull Team DNA Files**
```
1. Open Team DNA for both teams in matchup
2. Example: TOR @ FLA
   - Open toronto-maple-leafs.md
   - Open florida-panthers.md
```

**Step 3: Extract Key Intelligence**
```
For each team, note:
- Tier 1 players (bet every game)
- Tier 2 players (selective opportunities)
- Team offensive/defensive systems
- Situational factors (B2B, home/away)
- Goalie situation
```

**Step 4: Cross-Reference with Protocols**
```
- Apply APEXVIPER_GAME_ANALYSIS_PROTOCOL.md
- Check APEXVIPER_FATIGUE_MATRIX.md if B2B
- Use APEXVIPER_GOALIE_INTELLIGENCE.md for goalie matchup
- Reference shot-blocking-database.json for defensive impact
```

### Updating Team DNA Files

**When to Update:**
- New Tier 1/2 player identified
- Significant roster change (trade, injury)
- System change (new coach, strategy shift)
- Pattern emergence (3+ game trend)
- Monthly review and refresh

**Update Process:**
```
1. Open team DNA file
2. Update "Last Updated" date at top
3. Add new information in appropriate section
4. Maintain consistent formatting
5. Document source/evidence for changes
6. Commit with descriptive message
```
